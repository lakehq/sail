Temporary views backed by a data source, and related grammar pins.
All statements are plain SQL so the examples can be pasted into a Python shell.

The test data is written with pure SQL.

>>> spark.sql("""
...   INSERT OVERWRITE DIRECTORY '/tmp/sail_view_using/parquet_data'
...   USING parquet
...   SELECT * FROM VALUES (1, 'a'), (2, 'b') AS t(id, name)
...   """)
DataFrame[...]
>>> spark.sql("""
...   INSERT OVERWRITE DIRECTORY '/tmp/sail_view_using/csv_data'
...   USING csv OPTIONS (header 'true')
...   SELECT * FROM VALUES (1, 'a'), (2, 'b') AS t(id, name)
...   """)
DataFrame[...]
>>> spark.sql("""
...   INSERT OVERWRITE DIRECTORY '/tmp/sail_view_using/json_data'
...   USING json
...   SELECT * FROM VALUES (1, 'a'), (2, 'b') AS t(id, name)
...   """)
DataFrame[...]

`y` is a number literal suffix, so `1y` is a TINYINT literal, not an identifier.

>>> spark.sql("SELECT 1y").show()
+---+

`1m` is an identifier, not a literal. Spark and Sail have different errors
so we catch the exception.

>>> try:
...     error = spark.sql("SELECT 1m").collect()
... except Exception:
...     print("A column, variable, or function parameter with name `1m` cannot be resolved")
...
A column, variable, or function parameter with name `1m` cannot be resolved

Exceptions. Spark and Sail raise different error classes and messages for
these (Spark mostly fails at parse time, Sail in the analyzer), so we only
assert that each statement raises.
